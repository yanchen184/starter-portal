## 二、模組拆分與 SPI 解耦

### 2.1 拆分前的問題

所有認證邏輯（MOICA、OTP、Captcha、LDAP）都混在 `common-security-core` 裡，導致：
- 消費端無法選擇性引用（想用 OTP 就必須連 MOICA 一起裝）
- core 模組職責過多，難以維護
- 模組之間的依賴關係不清晰

### 2.2 拆分後的結構

```
care-security/
├── common-security-core              ← 核心（entity/repository/config/service）
├── common-security-auth-moica        ← 自然人憑證認證
├── common-security-auth-otp          ← TOTP 雙因子認證
├── common-security-auth-captcha      ← 圖形驗證碼
├── common-security-auth-ldap         ← LDAP / AD 認證
├── common-security-autoconfigure     ← 自動配置
├── common-security-spring-boot-starter ← Starter（全功能）
└── common-security-test              ← 整合測試
```

### 2.3 SPI 解耦

core 不能依賴 auth 子模組（否則循環依賴），透過 SPI 介面解耦：

```
core 定義介面（SPI）                 auth 子模組實作
──────────────────                 ─────────────
OtpChecker                        → OtpService implements OtpChecker
CaptchaVerifier                   → CaptchaService implements CaptchaVerifier
LdapAuthenticator                 → LdapAuthenticationProvider implements LdapAuthenticator
LdapUserSyncer                    → LdapUserSyncService implements LdapUserSyncer
```

**依賴方向永遠是單向的：**
```
auth-otp ──依賴──→ core（✅ 正確）
core ──依賴──→ auth-otp（❌ 不允許）
core ──依賴──→ OtpChecker（✅ core 自己定義的介面）
```

消費端不裝 `auth-otp`，OTP 功能就不存在（`ObjectProvider.getIfAvailable()` 回傳 null），系統照常運作。

### 2.4 Package 結構統一

所有 auth 子模組統一為 **package-by-layer**：

```
common-security-auth-moica/
├── controller/   CitizenCertController
├── service/      LoginTokenService, MoicaCertService, CitizenCertUserSyncService
├── util/         MoicaCertUtils, Pkcs7Utils
├── exception/    MoicaLoginException, ...
└── dto/

common-security-auth-otp/
├── controller/   OtpController
├── service/      OtpService, TotpService
└── dto/

common-security-auth-captcha/
├── controller/   CaptchaController
├── service/      CaptchaService
└── dto/

common-security-auth-ldap/
├── controller/   LdapController
├── service/      LdapUserSyncService
└── security/     LdapAuthenticationProvider
```

## 三、靜態分析（Checkstyle + SpotBugs）

### 3.1 架構

```
company-common-starters/
└── company-build-tools/                  ← 獨立模組（不繼承 root pom）
    └── src/main/resources/
        ├── checkstyle.xml                ← 程式碼風格規則
        └── spotbugs-exclude.xml          ← Bug 排除清單
```

任何專案只要在 `pom.xml` 加 plugin dependency 就自動套用：
```xml
<dependency>
    <groupId>com.company.common</groupId>
    <artifactId>company-build-tools</artifactId>
    <version>1.0.0</version>
</dependency>
```

### 3.2 Checkstyle 規則重點

| 規則 | 說明 |
|------|------|
| `AvoidStarImport` | 禁止 `import java.util.*` |
| `UnusedImports` | 禁止未使用的 import |
| `NeedBraces` | `if/for/while` 必須有大括號 |
| `EmptyCatchBlock` | 禁止空 catch（吞 exception） |
| `StringLiteralEquality` | 禁止字串用 `==` 比較 |
| `MethodLength` | 方法上限 80 行 |
| `ParameterNumber` | 方法參數上限 7 個（排除 constructor） |
| `ConstantName` | 常數全大寫（允許 `log`/`logger`） |
| `LineLength` | 行長度上限 150 |

**刻意不加的規則：**
- `LeftCurly`：Spring 的 `interface extends JpaRepository<X,Long> {}` 會誤報
- `UnnecessaryParentheses`：lambda 括號是偏好問題
- `Javadoc`：不強制，避免產生無意義的註解

### 3.3 SpotBugs 配置

- **effort: Max**（最大分析深度）
- **threshold: High**（只報高可信度 bug，減少誤報）
- 排除 JPA Entity、AutoConfiguration、Test 的已知誤報
- 排除 `TotpService.generateSecret()` 的 `SecureRandom` 誤報

### 3.4 觸發時機

```bash
mvn verify          # 自動執行 Checkstyle + SpotBugs，違規就 build 失敗
mvn checkstyle:check  # 單獨跑 Checkstyle
mvn spotbugs:check    # 單獨跑 SpotBugs
```

## 四、安全修復：OTP 暴力破解防護

### 4.1 問題

`POST /api/auth/otp/verify` 失敗時只回 400，**不觸發帳號鎖定**。

TOTP 是 6 位數字（100 萬種），`allowedSkew=1` 表示同時有 3 組有效 code，攻擊者可無限次嘗試。

### 4.2 修復

OTP verify 共用 `LoginAttemptService`，與密碼登入走同一套鎖定機制：

```java
// OtpController.verify()
boolean valid = otpService.verifyOtp(request.username(), request.code());
if (!valid) {
    loginAttemptService.loginFailed(request.username());   // 累積失敗次數
    return ResponseEntity.badRequest().body(ApiResponse.error("Invalid OTP code"));
}
loginAttemptService.loginSucceeded(request.username());     // 成功歸零
```

密碼錯 3 次 + OTP 錯 2 次 = 累積 5 次 → 帳號鎖定 30 分鐘。

## 五、分頁搜尋 API

### 5.1 通用 DTO

新增 `PageRequest`（`common-response-spring-boot-starter`），所有專案可共用：

```java
public class PageRequest {
    private int page = 0;        // 頁碼（從 0 開始）
    private int size = 20;       // 每頁筆數（預設 20，上限 100）
    private String sortBy;       // 排序欄位
    private String sortDir = "asc"; // 排序方向

    public int safePage() { ... }     // 不小於 0
    public int safeSize() { ... }     // 1 ~ 100
    public boolean isDescending() { ... }
}
```

搭配既有的 `PageResponse` 使用。

### 5.2 使用者搜尋 API

```
GET /api/users?keyword=王&page=0&size=20&sortBy=cname&sortDir=desc
GET /api/users?orgId=1&page=2&size=50
GET /api/users                         ← 預設：第 0 頁、20 筆、按 username 升序
```

回應格式：
```json
{
  "success": true,
  "data": {
    "content": [{ "id": 1, "username": "admin", ... }],
    "page": 0,
    "size": 20,
    "totalElements": 156,
    "totalPages": 8,
    "first": true,
    "last": false,
    "hasNext": true,
    "hasPrevious": false
  }
}
```

### 5.3 安全防護

- `size` 上限 100，防止一次撈太多導致 OOM
- `sortBy` 白名單驗證，只允許 `username / cname / email / enabled / lastLoginTime / accountLocked`
- 分頁查詢使用獨立 `countQuery`，避免 FETCH JOIN 導致的 count 問題

## 六、Controller 簡化

### 6.1 改動原因

原本每個 Controller 方法都手動包 `ResponseEntity<ApiResponse<>>`，但 `GlobalResponseAdvice` 已經會自動處理。

### 6.2 改動前後對比

```java
// Before（冗長）
@GetMapping("/{id}")
public ResponseEntity<ApiResponse<UserResponse>> findById(@PathVariable Long id) {
    return ResponseEntity.ok(ApiResponse.ok(userService.findById(id)));
}

// After（簡潔）
@GetMapping("/{id}")
public UserResponse findById(@PathVariable Long id) {
    return userService.findById(id);
}
```

回傳給前端的 JSON **完全一樣**。

### 6.3 規則

| 情境 | 寫法 |
|------|------|
| 正常回傳（200） | 直接 `return result` |
| 指定 status（201） | `@ResponseStatus(HttpStatus.CREATED)` |
| 動態 status（條件 400/404） | 保留 `ResponseEntity` |

### 6.4 影響範圍

10 個 Controller 全部簡化：AuthController、UserController、MenuController、RoleController、PermController、OrgRoleController、OtpController、CaptchaController、CitizenCertController、LdapController。

## 七、音訊驗證碼（Audio CAPTCHA）

### 7.1 背景

此功能參考 **Grails 舊系統**的驗證碼模組重新實作，使用純 Java 內建 API（`javax.sound.sampled`），不依賴外部 TTS 服務。目的是提供**無障礙（Accessibility）**支援，讓視障使用者也能通過驗證碼。

### 7.2 技術實作

`CaptchaService` 新增 `generateAudioBase64()` 方法：

- 每個字元（0-9、A-Z）對應不同頻率的 beep 音（C4 ~ B7 音階）
- 每字元 0.5 秒，字元間 0.2 秒間隔
- 正弦波生成，振幅 0.8（避免破音）
- 前後 5% 的 samples 做淡入淡出（避免爆音）
- 輸出：base64 WAV（16kHz, 16-bit, mono）

### 7.3 API

```
GET /api/auth/captcha/audio/{captchaId}    ← permitAll，不需認證
```

- `audioEnabled: true` 時回傳 WAV base64
- `audioEnabled: false`（預設）時回傳 404 + `"Audio CAPTCHA is not enabled"`

### 7.4 配置

```yaml
care:
  security:
    captcha:
      enabled: true
      audio-enabled: true    # 開啟音訊驗證碼
      chars: "ABCDEFGHJKMNPQRSTUVWXYZ23456789"   # 可配置字元集
      width: 160             # 圖片寬度
      height: 50             # 圖片高度
      font-size: 32          # 字型大小
```

### 7.5 與 Grails 舊系統的差異

| | Grails 舊系統 | Spring Boot 新系統 |
|---|---|---|
| 儲存 | Session | Redis（支援多實例部署） |
| 驗證 | 大小寫敏感 | 不分大小寫（`equalsIgnoreCase`） |
| 音訊 | 外部 TTS | 純 Java `javax.sound.sampled`（無外部依賴） |
| 字元集 | 固定 | 可配置（`chars` property） |
| 圖片尺寸 | 固定 | 可配置（`width` / `height` / `font-size`） |
| 一次性使用 | 否 | 是（驗證後立即從 Redis 刪除） |
