## 二、common-attachment-spring-boot-starter

### 2.1 背景

取代舊系統 `wez-attachment` Grails plugin。舊模組有多項架構問題：MIME 驗證可跳過、BLOB 全量載入記憶體、儲存模式硬切無抽象、下載無權限檢查、圖片壓縮同步阻塞。

### 2.2 核心功能

| 功能 | 說明 |
|------|------|
| 上傳 | Tika 偵測真實 MIME type + 檔案大小限制 + 26 種副檔名黑名單 + 路徑穿越防護 |
| 下載 | streaming 串流，不吃 heap |
| 軟刪除 | 繼承 BaseEntity，搭配 SoftDeleteRepository |
| 儲存策略 | filesystem / database blob 可切換（Strategy Pattern） |
| 存取控制 | AttachmentAccessPolicy 介面，消費端必須實作 |
| 圖片壓縮 | 上傳後異步壓縮（@TransactionalEventListener + Thumbnailator），壓縮後自動更新 DB fileSize |
| 事件驅動 | AttachmentUploadedEvent / AttachmentDeletedEvent（傳 ID，不傳 Entity） |

### 2.3 安全防護

| 防護 | 實作方式 |
|------|---------|
| MIME 偵測 | Apache Tika 讀取 magic bytes，不信任 client 提供的 contentType |
| 副檔名黑名單 | 26 種危險副檔名（.exe, .jsp, .php, .dll...） |
| 雙重副檔名攻擊 | `malware.exe.pdf` → 檢測到 `.exe.` 在中間 → 拒絕 |
| Trailing dot | `file.exe.` → Windows 會移除結尾 `.` 繞過檢查 → 拒絕 |
| 路徑穿越 | 拒絕 `..`、`/`、`\`、null byte + 儲存層 Path.normalize() 二次檢查 |
| 檔案大小 | 用實際 byte[] 大小驗證，不信任宣告的 fileSize |
| MIME 比對 | case-insensitive（`image/JPEG` 等同 `image/jpeg`） |
| 存取控制 | 預設 DefaultDenyAccessPolicy 全拒絕，消費端必須實作 |

### 2.4 上傳 Pipeline

```
Request
  → readAllBytes()             (避免 Tika 偵測後 stream 截斷)
  → Tika.detect()              (偵測真實 MIME type)
  → PathTraversalGuard         (order=50)
  → FileSizeValidator          (order=100)
  → MimeTypeValidator          (order=200)
  → StorageStrategy.store()
  → Persistence
  → AttachmentUploadedEvent    (commit 後觸發異步壓縮)
  → Response
```

### 2.5 Code Review 修正的 Critical Bugs

| Bug | 問題 | 修法 |
|-----|------|------|
| Stream 截斷 | Tika 偵測消耗 stream，存檔內容不完整 | 先 readAllBytes()，分別建 ByteArrayInputStream |
| 壓縮後 DB 不一致 | 圖片壓縮後 fileSize 沒更新 | 加 AttachmentRepository，壓縮後更新 entity |
| Detached Entity | Event 傳 JPA Entity，@Async handler 存取 lazy 屬性會爆 | 改傳 ID + 基本資訊 |
| @EnableAsync 全域副作用 | Starter 偷偷影響整個應用 | 移除，由消費端自行啟用 |

```

├── core/             ← AttachmentService（主 facade）, model/（DTO records）
```

### 2.8 消費端使用方式

```xml
<dependency>
    <groupId>com.company.common</groupId>
    <artifactId>common-attachment-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

消費端只需要：
1. **實作 `AttachmentAccessPolicy`** — 決定誰能存取/刪除
2. **設定 `application.yml`** — 儲存路徑、允許的 MIME type、檔案大小上限
3. **加 `@EnableAsync`** — 圖片壓縮需要（Starter 不會自動啟用，避免全域副作用）

### 2.9 設定範例

```yaml
wez:
  attachment:
    storage-type: filesystem
    storage-path: ./attachments
    max-file-size: 10MB
    allowed-mime-types:
      - image/jpeg
      - image/png
      - application/pdf
      - text/plain
    image:
      compression-enabled: true
      quality: 0.8
      compression-threshold: 500KB
```

## 七、技術決策記錄

### 7.1 為什麼上傳先 readAllBytes()？

Tika 偵測 MIME type 時會消耗 stream 的前 N 個 bytes。BufferedInputStream 的 mark/reset 在大檔案時不可靠。先讀成 byte[] 再分別建 ByteArrayInputStream 是最安全的做法。

### 7.2 為什麼 Event 傳 ID 不傳 Entity？

`@TransactionalEventListener(AFTER_COMMIT)` 在 commit 後執行，persistence context 已關閉。@Async handler 在另一個 thread 存取 detached entity 的 lazy 屬性會拋 LazyInitializationException。傳 ID 讓 handler 自己開 @Transactional 重新載入。

### 7.3 為什麼 Starter 不加 @EnableAsync？

@EnableAsync 是全域設定，會影響整個應用的行為。Starter 不應偷偷改變消費端的全域配置。

### 7.4 為什麼預設 Deny 而非 Allow？

每個專案的權限規則不同。如果預設 Allow，消費端忘記實作就是安全漏洞——任何人都能下載/刪除任何附件。預設 Deny + WARN log 提醒，確保不會默默放行。

### 7.5 為什麼 wez-email 和 wez-cache-redis 不做 Starter？

- wez-email 核心 31 行，只包裝 Grails Mail Plugin → `spring-boot-starter-mail` 直接取代
- wez-cache-redis 是 Spring Data Redis 薄包裝 → `spring-boot-starter-data-redis` 直接取代
- 做成 Starter 投入產出比太低

### 7.6 驗證碼語音檔大小評估

36 個 WAV 檔（0-9 + a-z），總計 1.1MB。每次語音驗證碼請求約 200KB（4 碼 WAV base64），等同一張小圖片，對前後端和網路都沒有負擔。
