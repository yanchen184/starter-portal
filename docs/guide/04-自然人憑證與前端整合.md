## 二、自然人憑證認證系統

### 2.1 什麼是自然人憑證

自然人憑證是台灣內政部憑證管理中心（MOICA）核發的數位身分證明，儲存在 IC 卡晶片中。它基於 **PKI（Public Key Infrastructure，公鑰基礎建設）** 架構，提供安全的線上身份認證。

#### 核心名詞解釋

| 名詞 | 白話解釋 |
|------|---------|
| **RSA-2048 金鑰對** | 一把「鎖」（公鑰）和一把「鑰匙」（私鑰）。用私鑰簽名的東西，只有對應的公鑰才能驗證，藉此證明「是你本人」。2048 是加密強度。 |
| **X.509 憑證** | 數位身分證。裡面包含：你的公鑰 + 身份資訊（身分證字號）+ 簽發機構。真實世界由 MOICA 簽發，存在 IC 卡晶片裡。 |
| **Challenge / Nonce** | 後端產生的一段隨機亂碼，只能用一次。防止重放攻擊（別人不能錄下你的簽章重複使用）。就像銀行說「請在紙上寫下 X7K9P2 然後簽名」，每次都不同。 |
| **SHA256withRSA** | 先用 SHA256 把資料壓成固定長度的「指紋」，再用 RSA 私鑰對指紋加密。合起來就是「我用私鑰對這段資料簽了名」。 |
| **JWT** | 登入成功後後端發的「通行證」。之後每次打 API 都帶著它，後端就知道你是誰。有過期時間。 |
| **PKCS#11** | 讀卡機和 IC 卡之間的通訊標準介面。 |
| **CRL / OCSP** | 憑證撤銷清單 / 線上憑證狀態協定。用來檢查憑證是否已被撤銷（例如卡片遺失掛失）。 |

### 2.2 認證流程

完整的 Challenge-Response 前後端交互流程：

```
步驟 1  [前端]           插入卡片
        讀卡機偵測到 IC 卡，讀取卡片內的憑證資訊。
        （Mock 版本：使用者輸入身分證字號，瀏覽器用 Web Crypto API 產生金鑰對 + 自簽憑證）

步驟 2  [前端]           產生金鑰對 + 憑證
        IC 卡晶片內已存有 RSA-2048 金鑰對和 MOICA 簽發的 X.509 憑證。
        （Mock 版本：前端現場產生，Subject DN = CN=身分證字號, O=MOICA, C=TW）

步驟 3  [前端 → 後端]    GET /api/auth/cert/challenge
        前端跟後端說：「我要登入，請給我一個挑戰碼。」
        後端產生 32 bytes 隨機 nonce，存入 Redis（TTL 300 秒），回傳 challengeId + nonce。

步驟 4  [後端 → 前端]    回傳挑戰碼
        前端收到 { challengeId: "uuid", nonce: "base64隨機碼" }。
        這個 nonce 有時效性（5 分鐘），且只能使用一次。

步驟 5  [前端]           IC 卡片簽章
        前端用私鑰對 nonce 進行 SHA256withRSA 簽章。
        真實情況：這一步發生在 IC 卡晶片內部，私鑰永遠不離開晶片。
        Mock 版本：瀏覽器用記憶體中的私鑰完成簽章。

步驟 6  [前端 → 後端]    POST /api/auth/cert/login
        前端將三樣東西打包送出：
        • challengeId — 回答的是哪一道題
        • certificate — Base64 編碼的 X.509 憑證（DER 格式）
        • signature   — Base64 編碼的數位簽章

步驟 7  [後端]           驗證
        後端依序執行：
        ① 用 challengeId 從 Redis 取出 nonce（原子性 getAndDelete，防重放）
        ② 解析 X.509 憑證，檢查有效期
        ③ 從憑證取出公鑰，驗證 SHA256withRSA 簽章
        ④ 從憑證 Subject DN 的 CN 欄位取出身分證字號
        ⑤ 查詢資料庫，若使用者不存在則自動建立（autoCreateUser）
        ⑥ 產生 JWT token（accessToken + refreshToken）
        ⑦ 記錄稽核日誌（IP、User-Agent、成功/失敗）

步驟 8  [後端 → 前端]    回傳 JWT
        驗證通過，回傳 TokenResponse：
        • accessToken（預設 30 分鐘）
        • refreshToken（預設 7 天）
        • authSource: "CITIZEN_CERT"
        • username、roles 等資訊
```

### 2.3 安全機制

| 機制 | 說明 |
|------|------|
| **私鑰不離開晶片** | 真實情況下，簽章在 IC 卡內部完成，私鑰無法被讀取或複製 |
| **一次性挑戰碼** | Redis 原子性 getAndDelete，每個 nonce 只能使用一次 |
| **挑戰碼過期** | TTL 300 秒（可設定），過期自動失效 |
| **數位簽章驗證** | SHA256withRSA，伺服器用憑證中的公鑰驗證 |
| **稽核日誌** | 每次登入嘗試（成功/失敗）都記錄 IP、User-Agent |

#### Mock 版本與真實版本的差異

| 項目 | 真實自然人憑證 | Mock 版本 |
|------|--------------|----------|
| 憑證簽發 | MOICA（政府 CA） | 瀏覽器自簽 |
| 信任鏈驗證 | Root CA → Sub CA → 使用者 | 不驗證 CA |
| 私鑰存放 | IC 卡晶片，無法讀取 | 瀏覽器記憶體 |
| 簽章位置 | IC 卡晶片內部 | Web Crypto API |
| 讀卡機 | 實體 USB + PKCS#11 | 按鈕模擬 |
| PIN 碼 | 需輸入 IC 卡 PIN 碼 | 無 |
| CRL/OCSP | 檢查憑證是否被撤銷 | 不檢查 |
| **Challenge-Response** | ✅ 一致 | ✅ 一致 |
| **SHA256withRSA** | ✅ 一致 | ✅ 一致 |
| **CN 取身分證字號** | ✅ 一致 | ✅ 一致 |

#### 我們少了什麼

**1. CA 信任鏈驗證**

真實系統中，憑證是由 MOICA（內政部憑證管理中心）這個政府 CA 簽發的。伺服器收到憑證後會驗證整條信任鏈：

```
MOICA Root CA（根憑證，預先安裝在伺服器上）
  └── MOICA Sub CA（中繼憑證）
        └── 使用者憑證（IC 卡裡的那張）
```

伺服器會逐層驗證：使用者憑證是不是 Sub CA 簽的？Sub CA 是不是 Root CA 簽的？只有整條鏈都通過才算合法。這能防止有人自己偽造一張憑證來騙過系統。

**我們的 Mock 版本跳過了這步** — 後端直接信任收到的任何憑證，只要簽章正確就通過。因為我們的憑證本來就是前端自簽的，沒有真正的 CA。

**2. CRL / OCSP 憑證撤銷檢查**

- **CRL（Certificate Revocation List，憑證撤銷清單）**：CA 定期公布一份「被撤銷的憑證清單」，伺服器下載比對。就像銀行的「掛失信用卡號碼清單」。
- **OCSP（Online Certificate Status Protocol，線上憑證狀態協定）**：伺服器即時向 CA 詢問「這張憑證還有效嗎？」，得到即時回覆。就像刷卡時 POS 機即時查詢銀行。

真實場景中，如果使用者的 IC 卡遺失並申報掛失，MOICA 會將該憑證加入撤銷清單。伺服器在驗證時會檢查，即使簽章正確也會拒絕。

**我們的 Mock 版本不做撤銷檢查** — 因為沒有真正的 CA，也就沒有撤銷機制。

**3. PKCS#11 介面**

PKCS#11 是一個密碼學標準介面（也叫 Cryptoki），定義了應用程式如何跟硬體安全模組（如 IC 卡、USB Token）溝通。真實流程：

```
應用程式 → PKCS#11 API → 讀卡機驅動 → USB 讀卡機 → IC 卡晶片
                                                      ↓
                                                 晶片內部完成簽章
                                                      ↓
應用程式 ← PKCS#11 API ← 讀卡機驅動 ← USB 讀卡機 ← 回傳簽章結果
```

重點是：**私鑰從頭到尾都在晶片裡面**，應用程式只是把「要簽的資料」送進去，拿到「簽好的結果」。PKCS#11 介面確保了這個流程的安全性和標準化。

**我們的 Mock 版本用瀏覽器的 Web Crypto API 代替** — 金鑰對存在瀏覽器記憶體中，簽章也在瀏覽器完成，沒有硬體保護。

**4. PIN 碼輸入**

真實的 IC 卡在使用私鑰前，會要求使用者輸入 PIN 碼（通常 6-8 位數字）。這是「你知道的」（PIN）+ 「你有的」（IC 卡）雙因素保護。連續輸錯 PIN 碼會導致 IC 卡鎖定，需要到戶政事務所解鎖。

```
使用者插入 IC 卡
      │
      ▼
系統要求輸入 PIN 碼
      │
      ▼
IC 卡晶片驗證 PIN ── 錯誤 → 錯誤次數 +1（通常 3-5 次後鎖卡）
      │
      ✅ 正確
      ▼
允許使用私鑰進行簽章
```

**我們的 Mock 版本沒有 PIN 碼** — 直接跳過這一步，按鈕點下去就簽章了。

#### 我們做對了什麼

核心的 **Challenge-Response 簽章驗證流程**是正確的，這是 PKI 認證的核心精神：

1. **Challenge-Response 機制** — 後端發隨機 nonce，前端簽章，後端驗證。這個模式不管是自然人憑證、健保卡、還是企業 PKI 都一樣。
2. **SHA256withRSA 簽章** — 跟真實憑證使用的演算法完全一致。
3. **X.509 憑證格式** — 我們產生的自簽憑證格式正確（ASN.1 DER 編碼），後端用標準 Java API 就能解析。
4. **CN 欄位取身分證字號** — MOICA 憑證確實把身分證字號放在 Subject DN 的 CN 欄位，我們的模擬方式正確。
5. **一次性 Nonce** — Redis 原子操作確保每個挑戰碼只能使用一次，跟真實系統的防重放機制一致。

**結論：** 我們的 Mock 版本在「認證流程的骨架」上是正確的。少的部分（CA 信任鏈、CRL/OCSP、PKCS#11、PIN 碼）都是「安全強度的加強」，而不是「流程的改變」。如果未來要對接真實的自然人憑證，後端只需要加上 CA 驗證和撤銷檢查，前端改接真實讀卡機 SDK，核心流程不需要改動。

#### Request / Response DTO

**CertLoginRequest（Record）：**

```java
public record CertLoginRequest(
    @NotBlank String challengeId,    // 挑戰碼 ID
    @NotBlank String certificate,    // Base64 X.509 憑證（DER）
    @NotBlank String signature       // Base64 數位簽章
) {}
```

**CertChallengeResponse（Record）：**

```java
public record CertChallengeResponse(
    String challengeId,  // UUID
    String nonce         // Base64 URL 編碼的 32 bytes 隨機碼
) {}
```

#### 錯誤處理

| 錯誤情況 | 異常類型 | HTTP 狀態 | 訊息 |
|----------|---------|-----------|------|
| Challenge 過期或已消費 | BadCredentialsException | 401 | "Challenge expired or already consumed" |
| 憑證過期或未生效 | BadCredentialsException | 401 | "Certificate is expired or not yet valid" |
| 簽章驗證失敗 | BadCredentialsException | 401 | "Invalid certificate signature" |
| 無法提取身分證字號 | BadCredentialsException | 401 | "Cannot extract citizen ID from certificate" |

#### 使用者自動建立

首次憑證登入時，`CitizenCertUserSyncService` 會自動建立使用者：

| 欄位 | 值 |
|------|---|
| username | 身分證字號（如 A123456789） |
| password | `{noop}CERT_MANAGED`（佔位，不使用密碼） |
| cname | 從憑證取得的顯示名稱 |
| citizenId | 身分證字號 |
| authSource | `CITIZEN_CERT` |
| 預設角色 | 依設定檔 `default-roles`（預設 ROLE_USER） |

### 2.5 設定屬性

```yaml
care:
  security:
    citizen-cert:
      enabled: true                    # 啟用自然人憑證認證
      challenge-expire-seconds: 300    # 挑戰碼 TTL（秒）
      auto-create-user: true           # 首次登入自動建帳
      default-roles:                   # 新建使用者的預設角色
        - ROLE_USER
```

所有 Bean 使用 `@ConditionalOnProperty(prefix = "care.security.citizen-cert", name = "enabled", havingValue = "true")` 條件載入，`enabled = false` 時完全不註冊任何 Bean。

## 三、圖形驗證碼前端整合

### 3.1 後端回顧

圖形驗證碼在 v0.0.3 已完成後端實作（詳見文件 03）。本階段完成前端整合。

**後端 API：**

| 方法 | 路徑 | 說明 |
|------|------|------|
| GET | `/api/auth/captcha` | 產生驗證碼，回傳 `{ captchaId, image }` |

**驗證碼特性：**
- 純數字 4 位，Redis 儲存，TTL 300 秒
- 一次性消耗：驗證後立即從 Redis 刪除（不管答對答錯都刪）
- 前端自適應：API 不存在時自動隱藏驗證碼欄位
- 隨機數使用 `SecureRandom`（密碼學安全等級）

### 3.2 圖片生成原理

使用**純 Java AWT** 自己繪製，沒有依賴任何第三方驗證碼套件（如 Kaptcha、Cage 等），保持最小化依賴。

#### 生成流程

```
① 建立畫布
   BufferedImage 160×50 px, RGB 色彩模式
   開啟 Anti-aliasing（抗鋸齒），讓文字邊緣平滑

② 填背景
   淺灰色 RGB(240, 240, 240)

③ 加干擾線（6 條）
   隨機顏色 RGB(0~199, 0~199, 0~199)
   隨機起點和終點
   隨機線寬 1~2px
   目的：干擾 OCR 辨識，增加機器破解難度

④ 加干擾點（30 個）
   2×2 px 的小方塊，隨機顏色，隨機散佈
   目的：進一步增加噪點干擾

⑤ 畫文字（每個數字獨立處理）
   字型：Arial Bold, 32pt
   顏色：隨機深色 RGB(0~100, 0~100, 0~100)，每個字元不同色
   水平位置：等距分佈 charWidth = 160 / (字元數 + 1)
   垂直位置：基準 35px ± 3px 隨機偏移
   旋轉：每個字元 ±0.2 弧度（約 ±11.5°）隨機旋轉
   旋轉後恢復，防止累積偏轉

⑥ 輸出
   PNG 格式 → Base64 編碼 → 作為 API 回應回傳
```

#### 圖片範例示意

```
┌──────────────────────────────────────────┐
│ ╱       ╲                   ╱            │  ← 干擾線（隨機方向）
│    · ·      3   7   ·  2      9    ·     │  ← 數字（隨機旋轉+偏移）+ 干擾點
│        ╲  ╱    ·         ╲     ╱         │  ← 干擾線
│   ·              ·                 ·     │  ← 干擾點
└──────────────────────────────────────────┘
  160px × 50px, 每個數字隨機顏色、隨機角度
```

#### 防破解機制

| 機制 | 說明 |
|------|------|
| 干擾線 | 6 條隨機線條穿過數字區域，干擾 OCR 分割 |
| 干擾點 | 30 個隨機噪點，增加背景複雜度 |
| 字元旋轉 | 每個數字 ±11.5° 隨機旋轉，破壞統一字型特徵 |
| 隨機顏色 | 每個字元顏色不同，干擾基於顏色的分割 |
| 位置偏移 | 垂直方向隨機偏移，防止固定座標切割 |
| 一次性消耗 | 即使答案被破解，也只能用一次 |
| SecureRandom | 密碼學安全的隨機數生成器，不可預測 |

#### 儲存與驗證

```
產生時：
  Redis Key:   captcha:{uuid}
  Value:       "3729"（正確答案）
  TTL:         300 秒

驗證時：
  ① 從 Redis 取出答案（get）
  ② 立即刪除（delete）— 不管對錯都刪，確保一次性
  ③ 比對使用者輸入（區分大小寫，trim 去空白）
  ④ 正確 → 繼續登入流程
     錯誤 → 拋出 IllegalArgumentException("Invalid captcha")
```

#### 技術選型理由

| 考量 | 說明 |
|------|------|
| **零外部依賴** | 只用 `java.awt` + `javax.imageio`，不引入額外套件 |
| **高度客製化** | 字型、大小、干擾程度、顏色範圍都可調整 |
| **輕量高效** | 純記憶體繪圖，生成一張圖片 < 5ms |
| **Starter 友好** | 不需要使用者額外安裝字型或套件，開箱即用 |

### 3.3 其他驗證碼生成方式比較

目前採用純 Java AWT 繪製。以下列出業界常見的替代方案及其優缺點，供未來升級參考：

#### 圖形驗證碼類

| 方案 | 套件 / 技術 | 優點 | 缺點 |
|------|------------|------|------|
| **純 Java AWT** (目前) | 無外部依賴 | 零依賴、輕量、完全可控、Starter 開箱即用 | 防破解能力一般，功能需自行擴充 |
| **Kaptcha** | `com.github.penggle:kaptcha` | Google 開源，支援漁網干擾、陰影、漸層背景、高度可設定 | 專案已停止維護（但穩定可用） |
| **Cage** | `com.github.cage:cage` | 極輕量（單一 JAR），API 簡單 | 功能較少，客製化有限 |
| **EasyCaptcha** | `com.github.whvcse:easy-captcha` | 支援 GIF 動畫驗證碼、算術題（如 3+5=?）、中文驗證碼 | 需額外依賴 |

#### 行為驗證碼類（使用者體驗更好）

| 方案 | 套件 / 技術 | 優點 | 缺點 |
|------|------------|------|------|
| **滑動拼圖驗證碼** | AJ-Captcha / 自製 | UX 好（拖動滑塊拼圖）、機器更難破解、支持行為分析 | 實作複雜度高，前後端都需配合 |
| **文字點選驗證碼** | AJ-Captcha | 點選指定文字順序，防 OCR 能力強 | 在手機上操作不太方便 |
| **旋轉驗證碼** | 自製 | 將圖片旋轉正確角度，直覺易用 | 需要準備圖片素材 |

#### 雲端服務類（防破解能力最強）

| 方案 | 服務商 | 優點 | 缺點 |
|------|--------|------|------|
| **Google reCAPTCHA v3** | Google | 無感驗證（使用者無需操作）、風險評分機制、最強防機器人 | 依賴外部服務、中國無法使用、隱私爭議 |
| **hCaptcha** | Intuition Machines | reCAPTCHA 替代品、注重隱私、GDPR 友好 | 依賴外部服務 |
| **Cloudflare Turnstile** | Cloudflare | 免費、無感驗證、不依賴 Cookie | 需要 Cloudflare 帳號 |

#### 選型建議

```
內部系統 / 低風險          → 純 Java AWT（目前方案）✅ 夠用
中風險 / 想提升 UX         → 滑動拼圖驗證碼（AJ-Captcha）
高風險 / 面對外部攻擊      → Google reCAPTCHA v3 或 Cloudflare Turnstile
特殊需求（算術/動畫/中文） → EasyCaptcha
```

目前方案對於內部管理系統而言已經足夠。如果未來面向外部使用者，建議升級為滑動驗證碼或雲端服務。

#### 滑動拼圖驗證碼（Slide CAPTCHA）技術原理

滑動拼圖驗證碼也叫 **Behavioral CAPTCHA（行為驗證碼）**，是目前主流的人機驗證方案。代表性實作有開源的 **AJ-Captcha**、**tianai-captcha**，以及商業服務 **GEETEST（極驗）**、**NetEase YIDUN（網易易盾）**。

**完整流程：**

```
[後端] 產生驗證碼
  ① 隨機選一張底圖（或使用預設圖庫）
  ② 隨機選一個 X, Y 座標作為正確位置
  ③ 從底圖中切出一塊拼圖形狀（如鑰匙孔形）
  ④ 產生兩張圖片：
     • 背景圖（有一個缺口）
     • 拼圖塊（要被拖到缺口的那塊）
  ⑤ 將正確 X 座標加密存入 Redis（不傳給前端）
  ⑥ 回傳：背景圖 + 拼圖塊 + 拼圖的 Y 座標（Y 座標可公開，只驗 X）

[前端] 使用者互動
  ① 顯示背景圖（有缺口）+ 拼圖塊（在左側）
  ② 使用者拖動滑桿，拼圖塊跟著水平移動
  ③ 前端記錄整個拖動過程的軌跡陣列：[{x, y, timestamp}, ...]
  ④ 使用者鬆手後，送出最終 X 座標 + 軌跡資料

[後端] 驗證（兩層防護）
  ① 座標驗證：比對 X 座標是否在正確位置 ±5px 以內
  ② 行為軌跡分析：判斷是真人還是機器人
```

**後端行為軌跡分析 — 這是防機器人的核心：**

前端不只回傳最終座標，還會回傳整個拖動過程的**軌跡陣列** `[{x, y, timestamp}, ...]`，後端分析以下特徵：

| 分析項目 | 真人特徵 | 機器人特徵 |
|----------|---------|-----------|
| **滑動速度** | 有加速→勻速→減速的過程 | 勻速直線，沒有變化 |
| **軌跡路徑** | Y 軸有微小抖動（手會抖） | 完美水平直線（Y 值不變） |
| **接近目標時** | 放慢速度，左右微調 | 一步到位，沒有修正 |
| **總耗時** | 1~5 秒 | < 0.5 秒，或固定時間 |
| **起步反應** | 有 0.2~0.5 秒延遲 | 立即啟動 |
| **軌跡點數量** | 數十到上百個點 | 極少的點（直接跳到終點） |

> **關鍵：** 即使機器人算出了正確的 X 座標，如果滑動軌跡太「完美」、太「規律」，後端也會判定為機器人並拒絕驗證。

**為什麼比圖形驗證碼更安全：**

```
圖形驗證碼的攻擊方式：
  OCR 辨識文字 → 破解率可達 80%+（靠 AI 模型）

滑動驗證碼的攻擊方式（需要同時攻破兩層）：
  ① 圖像分析找到缺口位置 → 困難但可行（靠 CNN 模型）
  ② 模擬真人滑動軌跡 → 非常困難（需要模擬物理運動特徵）
  兩者都要破才能通過，成本大幅提高
```

### 3.4 前端整合方式

登入頁面在載入時嘗試呼叫 `GET /api/auth/captcha`：
- **成功**：顯示驗證碼圖片 + 輸入框，登入時帶上 `captchaId` + `captchaAnswer`
- **失敗（404 或後端未啟用）**：隱藏驗證碼欄位，正常帳密登入

```typescript
// 前端自適應邏輯
const fetchCaptcha = async () => {
  try {
    const { data } = await axios.get<ApiResponse<CaptchaResponse>>(
      `${API_BASE_URL}/api/auth/captcha`
    );
    if (data.success && data.data) {
      setCaptchaEnabled(true);  // 顯示驗證碼
    }
  } catch {
    setCaptchaEnabled(false);   // 隱藏驗證碼
  }
};
```

### 3.5 使用者互動

- 點擊驗證碼圖片或「重新整理」按鈕可換一組新的驗證碼
- 登入失敗時自動重新整理驗證碼（因為舊的已被消費）
- 驗證碼輸入框限制為數字輸入

## 四、登入錯誤提示分類

### 4.1 後端錯誤分類

後端在登入流程中對不同錯誤拋出不同的異常：

| 錯誤場景 | 異常類型 | HTTP | 錯誤碼 | 訊息 |
|----------|---------|------|--------|------|
| 密碼錯誤 | BadCredentialsException | 401 | A0100 | "Invalid credentials" |
| 帳號不存在 | BadCredentialsException | 401 | A0100 | "Invalid credentials" |
| 驗證碼錯誤 | IllegalArgumentException | 400 | — | "Invalid captcha" |
| 驗證碼未填 | IllegalArgumentException | 400 | — | "Captcha is required" |
| 帳號鎖定 | LockedException | 423 | A0103 | "Account is locked..." |
| LDAP 帳號嘗試本地登入 | BadCredentialsException | 401 | A0100 | "Invalid credentials" |

> **安全設計：** 密碼錯誤與帳號不存在故意回傳相同訊息 `"Invalid credentials"`，防止帳號列舉攻擊（attacker 無法透過錯誤訊息判斷帳號是否存在）。

### 4.2 前端錯誤提示

前端根據後端回傳的訊息關鍵字進行分類顯示：

```typescript
const msg = getErrorMessage(err, '登入失敗');

if (msg.includes('captcha') || msg.includes('驗證碼')) {
  setError('驗證碼錯誤，請重新輸入');
} else if (msg.includes('credentials') || msg.includes('password')) {
  setError('密碼錯誤，請確認後重試');
} else if (msg.includes('locked')) {
  setError('帳號已鎖定，請稍後再試');
} else {
  setError(msg);
}
```

| 後端訊息 | 前端顯示 |
|----------|---------|
| "Invalid captcha" | 驗證碼錯誤，請重新輸入 |
| "Captcha is required" | 驗證碼錯誤，請重新輸入 |
| "Invalid credentials" | 密碼錯誤，請確認後重試 |
| "Account is locked..." | 帳號已鎖定，請稍後再試 |
| 其他 | 顯示原始訊息 |

### 4.3 錯誤提示流程圖

```
使用者按下「登入」
       │
       ▼
┌─ 驗證碼是否啟用？ ─┐
│  是                 │  否
▼                     ▼
驗證碼是否填寫？      直接送出帳密
│  否        │ 是     │
▼            ▼        ▼
提示：       送出帳密   後端驗證
請輸入驗證碼  + 驗證碼   │
             │         │
             ▼         ▼
        後端依序檢查：
        ① 驗證碼正確？ ─ 否 → 「驗證碼錯誤，請重新輸入」
        ② 帳號存在？   ─ 否 → 「密碼錯誤，請確認後重試」*
        ③ 帳號鎖定？   ─ 是 → 「帳號已鎖定，請稍後再試」

        ⑤ 全部通過     ─ 登入成功 → 進入系統

        * 故意與密碼錯誤顯示相同訊息，防帳號列舉
```

care:

    enabled: true

    # JWT 設定
    jwt:
      secret: your-256-bit-secret-key
      access-token-expire-minutes: 30
      refresh-token-expire-days: 7

    # 密碼策略
    password:
      min-length: 8
      require-uppercase: true
      require-lowercase: true
      require-digit: true
      require-special-char: false

    # 登入鎖定
    login-attempt:
      max-attempts: 5
      lock-duration-minutes: 30

### 7.3 啟用圖形驗證碼（可選）

```yaml
care:
  security:
    captcha:
      enabled: true           # 開啟驗證碼
      length: 4               # 數字長度
      expire-seconds: 300     # 過期秒數
```

**前端整合要點：**
- 呼叫 `GET /api/auth/captcha` 取得 `captchaId` + `image`（Base64 PNG）
- 登入時帶上 `captchaId` + `captchaAnswer`
- API 404 時隱藏驗證碼欄位（自適應）

### 7.4 啟用自然人憑證認證（可選）

```yaml
care:
  security:
    citizen-cert:
      enabled: true                    # 開啟憑證認證
      challenge-expire-seconds: 300    # 挑戰碼有效期
      auto-create-user: true           # 自動建帳
      default-roles:                   # 新使用者預設角色
        - ROLE_USER
```

**前端整合要點：**
1. 呼叫 `GET /api/auth/cert/challenge` 取得挑戰碼
2. 用私鑰對 nonce 進行 SHA256withRSA 簽章
3. 呼叫 `POST /api/auth/cert/login` 送出 `{ challengeId, certificate, signature }`
4. 成功回傳 TokenResponse（與密碼登入相同格式）

**需求：**
- Redis（儲存挑戰碼）
- 資料庫中有 `SaUser` 表（自動建帳用）

### 7.5 啟用 OTP 雙因素驗證（可選）
