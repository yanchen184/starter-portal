## 二、已完成項目詳情

### 2.1 POI XML 操作（Excel）

學習並實作的 POI 底層操作：

| 操作 | 用途 | API |
|------|------|-----|
| SXSSF 串流寫入 | EasyExcel 底層，大量資料不吃記憶體 | EasyExcel 封裝 |
| XSSF 後處理 | 在 EasyExcel 產出的 byte[] 上加 Pivot Table | `XSSFWorkbook` |
| CTDataFields XML | 同一欄位多個彙總函數時避免覆蓋 | `CTDataFields.addNewDataField()` |
| AreaReference | 定義 Pivot Table 的資料來源範圍 | `AreaReference` + `CellReference` |
| XWPFDocument | Word 圖片插入後處理 | `XWPFRun.addPicture()` |

**踩坑紀錄：**
- `XSSFPivotTable.addColumnLabel()` 對同一 column index 多次呼叫會覆蓋 → 改用 `CTDataFields` 直接操作
- POI 的 `cell.setText()` 會產生錯誤 XML（`<w:p><w:r><w:t>` 被當純文字寫入） → 改用 `cell.getParagraphs().getFirst().createRun().setText()`
- Pivot Table 沒有 cache data，Excel Online / LibreOffice 無法顯示 → 桌面版 Excel 可用「重新整理」

### 2.2 CSV 檔案格式

EasyExcel 原生支援，不需額外實作：

```java
ReportContext.builder()
    .engineType(EASYEXCEL)
    .outputFormat(OutputFormat.CSV)  // 直接切換
    .data(dataList)
    .build();
```

### 2.3 樞紐分析表（Pivot Table）

**PivotConfig DTO：**

```java
PivotConfig.builder()
    .sourceSheet("員工資料")
    .targetSheet("部門薪資分析")
    .row("部門")
    .value("薪資", SUM, "薪資總和")
    .value("薪資", AVERAGE, "平均薪資")
    .value("姓名", COUNT, "人數")
    .build()
```

**支援的彙總函數：** SUM / COUNT / AVERAGE / MAX / MIN / COUNT_NUMS

**實作原理：**
1. EasyExcel (SXSSF) 寫完資料 → byte[]
2. POI XSSF 打開 byte[]
3. 建立 Pivot Table Sheet（只記錄引用範圍，不複製資料）
4. 直接操作 CTDataFields XML 設定 value fields
5. 輸出最終 byte[]

**多 Pivot 支援：** 一個 ReportContext 可加多個 `.pivot()`

**效能：** 10 萬筆 + Pivot Table → 2.6 MB, ~8 秒

**限制：** 目前沒有填充 pivot cache，Excel Online / LibreOffice 需「重新整理」才能顯示。桌面版 Excel 打開即可。

### 2.4 DOCX 插入圖片

**兩種方式：**

| 方式 | 說明 | 範例 |
|------|------|------|
| 段落圖片 | 放在表格上方或文件任意位置 | `$logo` |
| 表格 cell 圖片 | 放在 1xN 表格的格子裡 | `$img1`, `$img2`, `$img3` |

**使用方式：**

```java
ReportContext.builder()
    .engineType(XDOCREPORT).outputFormat(DOCX)
    .templatePath("templates/report.docx")
    .image("img1", attachment1Bytes, 150, 80)
    .image("img2", attachment2Bytes, 150, 80)
    .image("img3", attachment3Bytes, 150, 80)
    .data(items)
    .build();
```

**實作：** 採用 POI 後處理（xDocReport 的 `addFieldAsImage` 對程式化範本不可靠）
1. xDocReport 處理 Velocity（`$img1` 不在 context 中，保留原文）
2. POI 打開結果 docx，找 `$img1` 等書籤文字
3. 清除書籤文字，用 `XWPFRun.addPicture()` 插入圖片
4. `removeEmptyTableRows()` 清除 `#foreach` / `#end` 殘留空行

### 2.5 DOCX/Excel 多資料集

**多 Sheet Excel：**

```java
.sheet("員工清單", employeeList)
.sheet("部門統計", deptList)
.pivot(PivotConfig.builder()
    .sourceSheet("員工清單").targetSheet("薪資分析")
    .row("部門").value("薪資", SUM, "薪資總和")
    .build())
```

**多 Context 合併：**

```java
ReportResult result = reportService.generate(ctx1, ctx2, ctx3);
```

各引擎合併策略：

| 引擎 | 策略 |
|------|------|
| EasyExcel | 每個 context 一個 Sheet |
| xDocReport | 合併 parameters + images，data 以 `items_0`, `items_1` 放入 |
| JasperReports | 合併多個 JasperPrint 的頁面 |

### 2.6 JasperReports 資料集模式

舊版（wez-report-ireport）用 SQL 查詢填充報表。新版改用 `JRBeanCollectionDataSource`：

```java
// 舊版：傳 SQL + JDBC Connection
JasperFillManager.fillReport(report, params, connection);

// 新版：傳 List 資料集
JRDataSource ds = new JRBeanCollectionDataSource(dataList);
JasperFillManager.fillReport(report, params, ds);
```

**好處：**
- 不需要在報表層存取 DB，關注點分離
- 資料可以在 Service 層做加工再傳入
- 支援任何 POJO / Map 作為資料來源
- 與 ReportEngine SPI 完美整合（統一用 ReportContext.data）

### 2.7 Redis 分散式限流

防止大量報表同時產製壓垮系統。使用 Redis INCR/DECR 實現跨機器的併發控制。

**兩層限流：**

| 層級 | Redis Key | 說明 |
|------|-----------|------|
| 全域 | `report:throttle:global` | 所有報表共用的同時產製上限 |
| 按名稱 | `report:throttle:name:{reportName}` | 每種報表的同時產製上限 |

**流程：**

```
acquire(reportName)
  → INCR global key → 超過？拋 429
  → INCR name key   → 超過？DECR global + 拋 429
  → 產製報表
release(reportName)
  → DECR name key
  → DECR global key
```

**防鎖死：** 所有 Redis key 設 TTL 10 分鐘，異常時計數器不會永久卡住。計數器歸零保護（`< 0` 時重設為 0）。

**設定：**

```yaml
common:
  report:
    throttle:
      enabled: true
      global-enabled: false          # 全域限流開關（預設關）
      global-max-concurrent: 10      # 全域同時產製上限
      default-max-concurrent: 10     # 每種報表預設上限
      limits:                        # 個別報表客製化上限
        員工清單: 5
        財務報表: 2
```

**使用方式：** 在 `ReportService.generate()` 中自動呼叫，消費端不需手動操作。透過 `ReportContext.reportName` 設定報表名稱（未設定則 fallback 到 fileName）。

### 3.2 DOCX → PDF 轉換

**確定方案：LibreOffice headless**

| 方案 | 品質 | 依賴 | 決策 |
|------|------|------|------|
| xDocReport converter | 差 | 已有依賴 | ❌ 複雜排版跑版、中文字體問題 |
| **LibreOffice headless** | 好 | 需裝 LibreOffice | ✅ **採用** |
| documents4j | 好 | Java wrapper + LibreOffice | 備選（底層同 LibreOffice） |
| Aspose.Words | 最好 | 商業授權 | ❌ 要付錢 |

**部署方式：**

```dockerfile
# Docker
RUN apt-get update && apt-get install -y libreoffice-writer --no-install-recommends
```

**呼叫方式：**

```java
ProcessBuilder pb = new ProcessBuilder(
    "soffice", "--headless", "--convert-to", "pdf", inputFile.getAbsolutePath(),
    "--outdir", outputDir.getAbsolutePath());
pb.start().waitFor(30, TimeUnit.SECONDS);
```

**注意事項：**
- 每次轉換啟動外部 process，約 1~3 秒/檔
- 並發需排隊（LibreOffice 同時只能跑一個 instance）
- 需 timeout 機制（LibreOffice 偶爾卡死）
- 如果只是要「產 PDF 報表」，直接用 JasperReports 品質更好，不需要 LibreOffice

### 3.3 Export 獨立 Server

**考量因素：**

| 面向 | 現狀（嵌入式） | 獨立 Server |
|------|---------------|------------|
| 部署 | 跟主應用一起 | 獨立部署、獨立擴展 |
| 資源 | 報表產製吃 CPU/記憶體影響主應用 | 隔離，不影響主應用 |
| 依賴 | JasperReports + POI 等大量 JAR 在主應用裡 | 主應用不需要這些依賴 |
| 複雜度 | 低 | 需要 API 通訊、佇列管理 |
| 適用場景 | 小型系統、報表量少 | 大型系統、報表量大、需排隊 |

**可能架構：**

```
主應用 → MQ（RabbitMQ/Redis） → Export Server → 檔案儲存（S3/DB）
                                                     ↓
                                              主應用查詢狀態/下載
```

**建議：** 目前用嵌入式就夠，非同步產製已經有排隊機制。等實際遇到效能瓶頸再拆分。

## 八、技術決策記錄

### 8.1 為什麼 Pivot Table 用後處理而非直接寫入？

EasyExcel 底層用 SXSSF（串流），不支援 Pivot Table API。POI XSSF 後處理只需打開 byte[]，加入 Pivot Sheet。Pivot Table 不複製資料，只記錄引用範圍。

### 8.2 為什麼 Pivot value fields 用 CTDataFields 而非 addColumnLabel？

POI 的 `addColumnLabel()` 對同一 column index 多次呼叫會覆蓋。直接操作 `CTDataFields` XML 可正確加入多個 value field。

### 8.3 為什麼圖片用 POI 後處理而非 xDocReport 的 addFieldAsImage？

xDocReport 的 `addFieldAsImage` 需要範本中有正確的 `<w:drawing>` XML 結構。用 POI 程式化產生的範本，純文字 `$logo` 不會觸發圖片插入。POI 後處理更可靠。

### 8.4 為什麼 xDocReport 的 #foreach 要放在獨立行？

`#foreach` 和資料放同一行會導致所有資料擠在同一個 cell。放獨立行後，xDocReport 正確識別資料行的 `<w:tr>` 並重複。空行由 `removeEmptyTableRows()` 後處理刪除。

### 8.5 為什麼 JasperReports 選 7.x？

JR 7.x 遷移到 Jakarta namespace，與 Spring Boot 4.x 相容。6.x 用 javax namespace 無法正常運作。jrxml 格式也從 XML Digester 改為 Jackson JSON-based。

### 8.6 為什麼 JasperReports 改用資料集而非 SQL？

傳 SQL 讓報表層直接存取 DB，職責不清。改用 `JRBeanCollectionDataSource` 讓 Service 層準備好資料再傳入，關注點分離、可測試性更好。

### 8.7 為什麼 Controller 不能放在 @ComponentScan 掃得到的包？

有 `@ConditionalOnProperty` 的模組，Controller 會被 `@ComponentScan` 繞過條件直接建立 bean。用 `excludeFilters` 排除或移到獨立包路徑。
