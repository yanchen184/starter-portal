# 團隊協作與知識管理 提案

## 核心理念

> **好的團隊，不應該因為少了誰就無法運作。**

知識如果只存在某個人的腦袋裡，那個人請假、換組、離職，工作就會停擺。
這不是個人能力的問題，而是組織缺乏知識管理的系統性風險。

我們要做的是：
- **需求規格、技術設計、決策脈絡，全部有文字紀錄**
- **新人看文件就能上手，不用靠老人口耳相傳**
- **不再有「這件事只有某某知道」的狀況**
- **人員異動不影響團隊的知識傳承與運作效率**

---

## 一、現況痛點

### 1.1 文檔與規格書
- 幾乎沒有 Spec 文件，需求靠口頭傳遞
- 技術文件散落各 Git Repo，難以搜尋
- 新人 onboarding 沒有參考資料，養成時間長
- 同樣的問題重複被問、重複踩坑
- PM 與工程師之間資訊落差大

### 1.2 溝通工具
- LINE 訊息容易被洗掉、無法追溯
- 無法針對單一議題進行討論（Thread）
- 新成員加入看不到歷史訊息脈絡
- 無法整合開發工具通知（Git、CI/CD）
- 重要決策散落在聊天記錄中

### 1.3 隱性成本
- 溝通成本：每人每天花 30~60 分鐘在「找資料」和「問人」，80 人團隊 = 每月損失 200+ 人時
- 重複犯錯：沒有知識沉澱，同樣的坑不同人重複踩，修一次 bug 可能要 2~4 小時
- 新人養成慢：沒有文檔支撐，onboarding 靠師徒制，新人從入職到產出的時間拉長 2~3 倍
- 決策不透明：重要脈絡隨人員異動而消失

## 二、知識管理平台需求

### 2.1 核心需求（P0）

#### 權限控管
- 依專案/部門分權，A 專案的人看不到 B 專案
- 依角色分權（PM / RD / QA / 主管）
- 可設定「全公司可見」與「專案內可見」兩種層級
- 由管理者指定誰能存取哪些空間

#### 內容模板
- 提供統一模板，新增文件時選模板照填即可
- 可設定必填欄位，確保內容完整度
- 需要的模板類型：
  - **PRD（需求規格書）**：PM 撰寫，描述需求背景、User Story、Acceptance Criteria
  - **TDD（技術設計文件）**：工程師撰寫，描述架構設計、資料流、API 設計
  - **API Spec**：工程師撰寫，描述 API 端點、參數、回應格式
  - **Onboarding 指南**：每個專案一份，新人看這份就能上手
  - **Meeting Notes**：會議決策紀錄，記錄什麼人、什麼時候、決定了什麼

### 2.2 重要需求（P1）
- **全文搜尋**：文件多了之後找得到才有用
- **版本歷史**：誰改了什麼、什麼時候改的、能回溯
- **協作編輯**：多人同時編輯，不用互傳檔案

### 2.3 加分需求（P2）
- **結構化組織**：樹狀目錄/空間/標籤，文件不會變垃圾場
- **整合能力**：跟 Git、Slack、CI/CD 串接
- **匯出功能**：能匯出 PDF/Word，給外部或非技術人員看

## 三、知識管理平台選項比較

| 平台 | 類型 | 權限控管 | 內容模板 | 全文搜尋 | 版本歷史 | 協作編輯 | PM 友善度 | 費用 (80人/月) |
|------|------|---------|---------|---------|---------|---------|----------|---------------|
| **Confluence** | SaaS | 強（空間/頁面/角色） | 強（模板+必填欄位） | 強 | 完整版本比對 | 有 | 中 | ~$580 USD |
| **Notion** | SaaS | 中（Workspace/頁面） | 強（Database+模板） | 強 | 有（免費版7天） | 有 | 高 | ~$800 USD |
| **Outline** | 自架/SaaS | 有（Collection 權限） | 有（模板功能） | 有 | 有 | 有 | 高（類似 Notion） | 免費（自架） |
| **BookStack** | 自架 | 有（書/章/頁權限） | 有 | 有 | 有 | 有限 | 高（WYSIWYG） | 免費（自架） |
| **GitBook** | SaaS | 有 | 有 | 有 | 有（Git 整合） | 有 | 中 | ~$500 USD |
| **VitePress** | 靜態站 | 無 | 弱（僅 Markdown 範本） | 有（已建置） | 靠 Git | 無（需會 Git） | 低 | 免費 |
| **Slite** | SaaS | 有 | 有 | 有 | 有 | 有 | 高 | ~$640 USD |
| **Coda** | SaaS | 有 | 強 | 有 | 有 | 有 | 高 | ~$720 USD |
| **MediaWiki** | 自架 | 有 | 有 | 有 | 有 | 有限 | 低 | 免費（自架） |

### VitePress 定位說明
VitePress 為靜態網站產生器，適合作為**公開技術文件站**，但缺乏權限控管、協作編輯、模板強制等能力，不適合作為全團隊的知識管理平台。現有的 starter-portal（VitePress）可繼續作為技術文件的公開展示層。

## 四、溝通平台需求

### 4.1 核心需求
- 訊息永久保留（不可有時間限制）
- 頻道/群組管理（依專案/團隊分頻道）
- Thread 討論（針對單一議題回覆，不會被洗掉）
- 全文搜尋歷史訊息

### 4.2 重要需求
- 整合開發工具（GitHub、CI/CD 通知）
- 檔案分享與保留
- 新成員加入可看到歷史訊息

## 五、溝通平台選項比較

| 平台 | 訊息保留 | Thread | 整合能力 | 費用 (80人/月) | 備註 |
|------|---------|--------|---------|---------------|------|

| **Slack Free** | 90 天 | 有 | 豐富 | 免費 | 訊息 90 天限制，不適合長期使用 |
| **Mattermost** | 無限（自架） | 有 | 不錯 | 伺服器成本 | 開源，類似 Slack，需維運 |
| **Discord** | 無限 | 有 | 有限 | 免費 | 企業觀感不正式 |
| **Google Chat** | 需 GWS | 有 | Google 生態系 | 需購買 GWS ~$960 USD | 需要 Google Workspace 授權 |
| **Microsoft Teams** | 無限 | 有 | Office 整合 | 需購買 M365 ~$480 USD | 公司目前無 M365 |

## 六、組合方案

### 方案 A — 零成本自架方案
| 項目 | 選擇 |
|------|------|
| 知識管理 | Outline（自架） |
| 溝通平台 | Mattermost（自架） |
| 技術文件 | starter-portal（VitePress，已有 POC） |
| 月費 | 僅伺服器成本 |
| 優點 | 零授權費、資料完全自控 |
| 缺點 | 需要維運人力、生態系較小 |

### 方案 B — 快速導入方案
| 項目 | 選擇 |
|------|------|
| 知識管理 | Notion |
| 溝通平台 | Slack Pro |
| 技術文件 | starter-portal + Git → Notion 同步 |
| 月費 | ~$1,400 USD |
| 優點 | 上手最快、PM 友善度最高、零維運 |
| 缺點 | 費用較高、Notion 權限控管不如 Confluence |

### 方案 C — 企業級方案
| 項目 | 選擇 |
|------|------|
| 知識管理 | Confluence |
| 溝通平台 | Slack Pro |
| 技術文件 | starter-portal + Git → Confluence 同步 |
| 月費 | ~$1,170 USD |
| 優點 | 權限最完整、企業功能最強、可整合 Jira |
| 缺點 | 學習曲線稍高、介面較重 |

## 七、推動計畫（不論選哪個方案）

### Phase 1 — 試行期（第 1~2 個月）
- 選定 1 個專案團隊（5~10 人）先行試用
- 建立溝通平台 Workspace，設定基本頻道
- 在知識平台上建立文件模板（PRD、TDD、API Spec）
- starter-portal 作為技術文件 POC 持續運作

### Phase 2 — 擴展期（第 3~4 個月）
- 根據試行回饋調整流程與模板
- 擴展到 2~3 個團隊
- 建立文件撰寫規範 & 最佳實踐
- 整合 Git / CI 通知到溝通平台

### Phase 3 — 全面導入（第 5~6 個月）
- 全團隊導入
- 建立 onboarding 文件流程
- 定期 review 文件品質
- 評估是否需要調整方案或升級
